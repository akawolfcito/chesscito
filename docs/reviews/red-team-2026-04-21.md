# Red Team Report — 2026-04-21

Adversarial audit of the play-hub, arena, and destination-sheet flows. Driven by direct Playwright captures against the live production URL (`https://chesscito.vercel.app`) on iPhone 13 (390×844 @3x, WebKit UA) and desktop (1280×800, Chromium). Code-side flows traced against `main` at commit `0c0f858`.

The session-long dock + board saga is already documented in individual commits; this report covers the **remaining** adversarial findings — flows that are structurally fragile, dead-end candidates, state leaks, and race conditions that are NOT yet blocked by tests.

## Broken Flows

None observed on core navigation paths after the commits from this session. The board-0×0 flow that was broken on all viewports is now fixed and guarded (see `e2e/surface-integrity.spec.ts`).

## Dead Ends

### D1. `handleClaimVictory` insufficient-balance path

**Where:** `app/arena/page.tsx:373-543` — the Victory NFT claim flow.

**Trigger:** User wins on /arena, taps "Mint Victory" with a wallet that has 0 USDC/cUSD.

**Symptoms:** `selectPaymentToken(mintPriceUsd6)` returns `null` at line 402, which throws `"No token with sufficient balance"`. The `catch` at line 523 sanitizes it to `"Insufficient balance"` and sets `claimPhase = "error"`. The `<ArenaEndState>` overlay surfaces the error. User has to manually close and leave — no "top up wallet" CTA or link to the shop.

**Severity:** Dead end for free users who read the victory screen. Not game-breaking (they can still Play Again), but the minting flow silently ends.

**Suggested guardrail:** E2E spec that mints with an empty-wallet fixture and asserts the error message is actionable (links to ramp / shop).

### D2. `setCoachPhase("paywall")` loop if `selectPaymentToken` returns null

**Where:** `app/arena/page.tsx:302-369` — `handleBuyCredits`.

**Trigger:** User on the coach paywall sheet taps Buy with no token balance.

**Symptoms:** `selectPaymentToken(priceUsd6)` returns null at line 306, `setCoachPhase("idle")` fires at 308, and the flow silently drops. The paywall closes with no feedback. User thinks nothing happened.

**Severity:** Soft dead end. Easy to recover (reopen paywall) but confusing.

**Suggested fix:** toast/error message on the paywall before closing.

## State Leaks

### S1. `autoResetTimer.current` partially covered by cleanup

**Where:** `app/page.tsx` — `autoResetTimer` ref.

**Trigger:** Fast board navigation during a pending reset.

**Symptoms:** There's ONE cleanup effect at page.tsx:198-202 that clears the timer on unmount. But during *in-session* navigation (setSelectedPiece), the timer is cleared inline at 789. If a failure phase starts an auto-reset and the user changes piece mid-timer, the cleanup at 789 fires — good. But if the user hits back-to-hub from arena while a reset timer is still pending on play-hub (un-mounted), the cleanup effect should handle it. This looks OK but is two-source-of-truth — a refactor moving reset logic could break it.

**Severity:** Latent. No current bug, but fragile.

**Suggested guardrail:** centralize auto-reset clear in a single custom hook with its own cleanup.

### S2. `sessionStorage["chesscito:claim"]` may outlive a tab close mid-claim

**Where:** `app/arena/page.tsx:399` (write) and `app/arena/page.tsx:104-126` (read on mount).

**Trigger:** User starts a claim (fetches signature + sets sessionStorage to `"claiming"`), closes the tab before the mint tx is sent.

**Symptoms:** On return, `chesscito:claim` has `phase: "claiming"`. The mount-time hook at line 118-124 detects that and clears the key — good, that's the intended defensive path. But the deadline isn't re-checked; if it was a long claim with a future deadline, we lose the opportunity to resume.

**Severity:** Minor UX — user just has to re-initiate the mint. No funds lost.

### S3. Coach `coachAbortRef.current` leak across rapid "Ask Coach" clicks

**Where:** `app/arena/page.tsx:182-259` — `startCoachAnalysis`.

**Trigger:** User spams the "Ask Coach" button during a long analysis.

**Symptoms:** Line 187 aborts the previous controller before creating a new one. That works. But the fetch resolves on abort with an error, and the catch at line 248 checks `signal.aborted` → early return. Correct.

**Severity:** None — handled correctly.

## Race Conditions

### R1. Two `writeContractAsync` calls during rapid wallet reconnect

**Where:** `handleClaimVictory` + `handleBuyCredits`.

**Trigger:** Wallet disconnect/reconnect mid-approval. `prevConnected.current` logic at page.tsx:143-151 resets claim error but NOT the `claimingRef.current` guard.

**Symptoms:** If the user had a claim in flight and the wallet reconnects, a second attempt can start before the first's `finally` clears `claimingRef`. Double-submit guard exists (`claimingRef.current = true` at line 376 + check at 375), so this should reject a rapid second click. But the wallet-reconnect-triggered `setClaimPhase("ready")` could bypass the UI disabled state briefly.

**Severity:** Low — the mint contract also has nonce-based replay protection, so double-mint is impossible.

**Suggested guardrail:** explicit test covering mid-claim disconnect/reconnect.

### R2. `useChessGame`'s `aiTimeoutRef` vs reset race

**Where:** `lib/game/use-chess-game.ts:56, 140-185`.

**Trigger:** Rapid "Play Again" during AI thinking.

**Symptoms:** AI move is scheduled via `setTimeout(..., 50)`. If user hits reset while the 50ms timer is pending, the callback fires on the old `gameRef.current` state and may try to move a piece that was just reset. Not breaking because `game.isGameOver()` short-circuits, but could emit state for a stale game.

**Severity:** Latent.

**Suggested fix:** clear `aiTimeoutRef` in the reset function. Already cleared in the useChessGame's reset — verified.

## Edge Cases

### E1. /arena reached without wallet connected

**Where:** Arena gate + Victory mint.

**Behavior observed via Playwright:** Arena page loads fine without wallet. "Enter Arena" works. Game plays. On win, "Mint Victory" shows — but the button is disabled/hidden when `!isConnected || !isCorrectChain`. User sees the victory state and the Play Again CTA, with a "Connect Wallet" prompt elsewhere.
