# Chesscito UX Writing Brief v1

Status: Approved editorial source of truth for visible product copy, badge metadata, CTA language, onboarding/tutorial copy, move feedback, score flow, claim flow, and other key user-facing states.

Audience:
- Product
- Design
- Engineering

Scope:
- Visible product copy
- Badge metadata
- CTA language
- Onboarding/tutorial copy
- Move feedback
- Score flow
- Claim flow

## Voice and tone

Chesscito should sound:
- clear
- subtly magical
- stimulating

### Interpretation

- Clear: the user should understand what to do without friction.
- Subtly magical: the product may feel enchanting, but never obscure.
- Stimulating: the copy should create energy and forward motion.

## Localization

- Visible product: English-first
- Spanish may exist later as a secondary editorial layer for community, campaigns, or external content
- Onchain metadata: English-only

## What happens

The user is mainly trying to:
- learn a pattern
- complete a piece path
- claim a badge
- submit a score

## Who it happens to

Primary user:
- MiniPay / Celo user

### Working assumption

This user may be:
- mobile-first
- curious
- not deeply familiar with chess terminology
- expecting fast clarity and lightweight progression

## When it happens

These are the key product moments where copy matters most:
- during first entry
- before the first move
- after a wrong move
- after a correct move
- when a piece path is completed
- when a badge becomes claimable
- before submitting score
- when progress feels unclear

## Where it impacts

Copy should be designed by product surface:
- screen
- component
- state
- event
- metadata

Examples:
- onboarding screen
- tutorial panel
- move feedback state
- badge claim modal
- score submission flow
- NFT metadata

## Why we make this product

Chesscito exists to make chess learning feel playable, accessible, and visibly progressive through a mobile-first experience.

## How we want to create impact

We want to:
- reduce friction while learning
- transform practice into visible progress
- connect learning with collectible identity

## CTAs

### Principles

- short
- direct
- action-first
- one clear intention per screen
- no ambiguity

### Example CTA set

- Start Trial
- Make Your Move
- Try Again
- Continue
- Claim Badge
- Submit Score

## Pain points we want to address

- "I don't know what to do."
- "I don't understand why I failed."
- "I don't feel progress."
- "The app talks to me like I'm already an expert."
- "Everything sounds generic or emotionally flat."

## First 30-second clarity goal

In the first 30 seconds, the player should understand:
- what this challenge is
- what piece they are learning
- what success looks like
- what to do next

## What we should never say

We should avoid:
- humiliating or punishing language
- unnecessary technical jargon
- inflated claims
- crypto-native language that creates confusion
- long explanations for simple actions
- wording that implies financial rewards if the product does not guarantee them

### Never imply

- guaranteed earnings
- token rewards
- financial upside
- prize-like expectations unless explicitly real and available

## Emotions we want to create

- curiosity
- clarity
- momentum
- satisfaction
- pride in progress

## Microcopy principles

- one idea per line or sentence
- prefer verbs
- keep instructions short
- explain the action before the system
- use the same term consistently across UI
- teach without sounding academic
- encourage without sounding childish
- keep the magical tone subtle, never theatrical

### Good direction

- "Complete the Bishop path."
- "Try again."
- "Badge ready to claim."

### Avoid

- "Embark upon your glorious mastery journey."
- "You have failed this sacred challenge."
- "Execute the bishop movement validation."

## Naming system

### Core rules

- use English-first naming
- use piece names in English
- avoid "Master"
- use "Ascendant" as the current rank direction
- prefer concise, badge-friendly naming

### Current badge titles

- Rook Ascendant
- Bishop Ascendant
- Knight Ascendant

### Preferred system structure

[Piece] + Ascendant

## Glossary and fixed terms

Use these terms consistently across the app:
- Badge
- Claim Badge
- Submit Score
- Piece Path
- Trial
- Progress
- Rook Ascendant
- Bishop Ascendant
- Knight Ascendant

Do not alternate casually between:
- badge / achievement / collectible / reward
- lesson / level / stage / mission
- score / result / ranking / points

Choose one term per concept and keep it stable.

## Editorial decisions already approved

- English-first for visible product copy
- English-only for onchain metadata
- Ascendant as the base badge rank naming
- Current badge set:
  - Rook Ascendant
  - Bishop Ascendant
  - Knight Ascendant

## Execution rules

1. This brief is the editorial source of truth for product-facing copy and onchain metadata naming.
2. New user-facing copy should default to English unless and until a formal localization layer exists.
3. Badge metadata should remain English-only, including `name`, `description`, and attribute labels.
4. Product surfaces should favor consistency over novelty. If a concept already has an approved term in this brief, reuse it.
5. Copy updates should be audited by surface:
   - screen
   - component
   - state
   - event
   - metadata
6. Avoid broad rewrites without first checking for glossary and CTA consistency.
